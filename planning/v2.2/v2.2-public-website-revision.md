# Evolvo v2.2 Public Website Revision

## Purpose

This document defines the Evolvo v2.2 revision that adds a public website layer to Evolvo.

It is a follow-up revision to the existing v2 and v2.1 planning pack.

The purpose of v2.2 is to add a public-facing marketing website and blog to the same Next.js application, while keeping the actual product protected behind authentication.

This document should be read as a delta on top of the existing planning files.

---

# Summary of the v2.2 change

Evolvo already has a product application and authenticated operator experience.

Evolvo v2.2 adds:

- a public homepage
- a public blog
- public blog post pages
- clear protected product routes
- an explicit login route for internal/operator access
- an admin bootstrap mechanism for initial access setup

The public website is not a signup funnel yet.

There should be:
- no public signup flow
- no signup language
- no “start free” CTA
- no requirement to mention login anywhere on the public site

The login route exists for operator access only.

---

# Core v2.2 rules

## Rule 1 - Same Next.js application

The public website and the protected product UI must live in the same Next.js app.

There is no separate marketing app in v2.2.

## Rule 2 - Public and protected routes must be explicit

The system must clearly separate:

- public routes
- protected product routes

Recommended structure:

- `/`
- `/blog`
- `/blog/[slug]`
- `/login`
- `/app/...`

This separation should be reflected in routing, layout structure, and access control.

## Rule 3 - Homepage is static-first

The homepage should be implemented as static content.

It should not depend on the API for first release.

The homepage is a public marketing surface describing Evolvo as an autonomous software factory.

## Rule 4 - Blog is repo-backed content

The blog should be powered by local markdown or MDX files stored in the repository.

The blog is not API-backed in v2.2.

The blog is not CMS-backed in v2.2.

## Rule 5 - Login exists but is not promoted publicly

The `/login` route must exist and work, but should not be linked or advertised on the public site.

There should be no public mention of:
- signing up
- creating an account
- requesting access
- free trials
- pricing conversion CTAs

## Rule 6 - Product routes must be protected

The real product/operator UI must sit behind protected routes.

Unauthenticated users must not be able to access protected product pages.

Authentication and route protection must be explicit and durable.

## Rule 7 - Admin bootstrap must exist

There must be a script that can be run to set or reset the admin user email and password.

This is required for initial system access and operational recovery.

# Public website scope

## Public routes

The following routes are public in v2.2:

- `/`
- `/blog`
- `/blog/[slug]`

These routes must be accessible without authentication.

## Protected routes

The product area should live under:

- `/app`

Examples:
- `/app`
- `/app/projects`
- `/app/projects/[id]`
- `/app/settings`
- `/app/runtime`
- `/app/usage`

These routes must require authentication.

## Login route

The following route should exist:

- `/login`

This route is public in the sense that it must be reachable without an active session, but it is not part of the marketed route structure.

It should not be surfaced in the public navigation.

- autonomous software factory

It should communicate:
- what the product is
- what it does
- who it is for
- why it is valuable

# Blog requirements

## Blog model

The blog should be powered by markdown or MDX files in the repo.

Each post should have frontmatter metadata.

## Public blog routes

Required routes:
- `/blog`
- `/blog/[slug]`

## Blog behaviour

The blog should support:
- public post listing
- public post detail pages
- metadata per post
- SEO-friendly routes
- static generation where appropriate

Optional but recommended:
- tags
- categories
- RSS
- sitemap inclusion
- Open Graph metadata

## Initial content strategy

Posts are authored locally in the repo.

No CMS is required for v2.2.

No API content dependency is required for v2.2.

---

# Route-protection model

## Public area
Public routes must remain accessible without authentication.

## Protected area
Protected product routes must sit under `/app` and must require authentication.

## Login flow
If an unauthenticated user tries to access a protected route:
- redirect to `/login`
- optionally preserve intended destination for post-login redirect

## Session handling
Authenticated users should be able to access:
- protected product routes
- operator tools
- admin-only areas where permitted by role

Public pages should remain public regardless of session state.

---

# Authentication implications

## Existing auth
If auth already exists, v2.2 should formalise route protection and login behaviour.

## Admin bootstrap
There must be a script that can:
- create an admin if none exists
- update admin email and password
- reset admin credentials safely when needed

The exact implementation may vary, but it should be:
- local and explicit
- safe to run intentionally
- documented

# SEO and discoverability implications

## Homepage
The homepage should have:
- good metadata
- strong title and description
- Open Graph support
- clear semantic structure

## Blog
Blog posts should have:
- per-post metadata
- canonical route shape
- indexability
- good heading structure

# Architecture implications

## Same app, separate concerns

The same Next.js app now serves two concerns:

### Public website
- read-heavy
- public
- static-first
- marketing and content

### Protected product UI
- authenticated
- app-like
- data-heavy
- API-driven with TanStack Query

This split must be kept clear in implementation.

## Public content boundary

Public homepage and blog content should not introduce unnecessary API coupling.

The protected product UI continues to use the API as before.

---

# Runtime and API implications

The public website does not materially change runtime responsibilities.

The API may support auth/session handling for login and protected routes, but the homepage and blog should not require the API for their primary content in v2.2.

The runtime is unaffected by the public website addition.

# Benefits of the v2.2 revision

This revision improves the system by:

- giving Evolvo a public-facing presence
- clarifying public vs protected route boundaries
- adding SEO and content capabilities
- allowing product communication without exposing signup
- keeping the real product behind authentication
- supporting internal/operator access through a stable login flow
- adding an admin bootstrap path for reliable access management
