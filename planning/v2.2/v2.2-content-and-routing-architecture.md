# Evolvo v2.2 Content and Routing Architecture

## Purpose

This document defines the content structure, route architecture, and public-content model for the Evolvo v2.2 website layer.

It complements the v2.2 public website revision and backlog revision docs.

---

# Route architecture

## Public routes

The following routes are public:

- `/`
- `/blog`
- `/blog/[slug]`

These routes should render without authentication.

## Access route

The following route exists for operator access:

- `/login`

This route is reachable without authentication but is not part of public marketing navigation.

## Protected product routes

The protected product UI should live under:

- `/app`

Examples:
- `/app`
- `/app/projects`
- `/app/projects/[id]`
- `/app/runtime`
- `/app/usage`
- `/app/settings`

These routes must require authentication.

---

# Route-grouping recommendation

A clean structure in Next.js should separate:

- public website routes
- auth access routes
- protected product routes

Recommended conceptual split:

- public layout
- auth layout
- app layout

The exact implementation may use route groups, layout nesting, middleware, or guard wrappers as appropriate.

The key requirement is clarity of access boundaries.

---

# Navigation model

## Public navigation

Public navigation should be minimal.

Recommended items:
- Home
- Blog

Do not include:
- Login
- Signup
- Pricing if it does not exist yet
- Internal app links

## Protected navigation

Protected app navigation should serve operator and product workflows and should remain separate from public marketing navigation.

# Homepage architecture

## Homepage objective

The homepage should explain Evolvo as an autonomous software factory.

It should communicate the value, workflow, and positioning of the product without requiring public signup flows.

## Recommended homepage sections

### 1. Hero
Purpose:
- explain what Evolvo is
- establish positioning immediately

Suggested content intent:
- headline
- short supporting copy
- one or two product framing statements
- restrained CTA pattern, likely to content rather than signup

### 2. Product positioning
Purpose:
- explain the category and the problem it solves

Suggested content intent:
- what an autonomous software factory means
- what Evolvo automates
- where human control still exists

### 3. How it works
Purpose:
- explain the operator-approved execution model

Suggested content intent:
- product description plus development plan
- planning expansion
- approval gate
- dev, review, release execution
- controlled human intervention

### 4. Core capabilities
Purpose:
- communicate practical product value

Suggested content intent:
- structured planning
- execution lanes
- isolated worktrees
- review and release flow
- observability and control plane

### 5. Architecture and trust layer
Purpose:
- explain why the system is credible and controlled

Suggested content intent:
- hosted control plane
- local runtime execution
- approval-first workflow
- protected operational model

### 6. Blog preview
Purpose:
- surface writing and give the homepage a living content layer

Suggested content intent:
- latest 3 posts
- title, date, description
- link to full blog index

### 7. Footer
Purpose:
- simple site closure

Suggested content intent:
- copyright
- perhaps blog link
- perhaps product one-liner

# Homepage content rules

## What the homepage should do
- explain the product
- build credibility
- support SEO
- support reading and discovery
- set product framing clearly

## What the homepage should not do
- mention signup
- push self-serve onboarding
- promise public availability if that is not true
- expose internal implementation details in a confusing way
- mention `/login`

# Blog architecture

## Content source

The blog should be powered by local markdown or MDX files stored in the repo.

Recommended content directory:

- `apps/web/content/blog/`

Alternative acceptable structure:
- `content/blog/`

The important rule is that content is local, versioned, and repo-backed.

## Content format

Each blog post should include frontmatter.

Recommended required fields:
- `title`
- `description`
- `slug`
- `publishedAt`

Recommended optional fields:
- `updatedAt`
- `tags`
- `author`
- `coverImage`
- `draft`

## Example frontmatter shape

```md
---
title: "How Evolvo Plans Software Delivery"
description: "How planning expansion, approval, and execution work in Evolvo."
slug: "how-evolvo-plans-software-delivery"
publishedAt: "2026-03-10"
updatedAt: "2026-03-10"
tags: ["planning", "agents", "software-factory"]
draft: false
---
````

## Blog index page

The blog index should:

* list published posts
* sort descending by publish date
* render title, description, and date
* support clean navigation to detail pages

## Blog detail page

The blog detail page should:

* resolve by slug
* render markdown or MDX content
* expose metadata
* support headings and content formatting
* support good reading layout

---

# Content-processing requirements

## Parsing

The blog pipeline should:

* read files from repo content directory
* parse frontmatter
* validate required fields
* ignore or exclude drafts where appropriate

## Rendering

Rendering should support:

* markdown at minimum
* MDX if desired

## Static generation

Public blog routes should be statically generated where practical.

Homepage should also be static-first.

# SEO architecture

## Site-level metadata

The public site should define:

* default title
* default description
* Open Graph defaults
* canonical base URL

## Homepage metadata

The homepage should have:

* product-focused title
* product-focused description
* Open Graph metadata

## Blog index metadata

The blog index should have:

* blog-specific title
* blog-specific description

## Blog post metadata

Each post should have:

* title
* description
* canonical URL
* Open Graph metadata
* article-like semantics where relevant

## Sitemap

The sitemap should include:

* `/`
* `/blog`
* all public blog post routes

It should exclude:

* `/app/...`
* operator-only routes
* any internal-only admin paths

## RSS

An RSS feed is recommended for the blog.

# Protected-route architecture

## Intent

Protected routes exist for the real product UI and should be treated as application surfaces, not public website surfaces.

## Behaviour

Unauthenticated request to `/app/...` should:

* redirect to `/login`

Authenticated request to `/app/...` should:

* render the product UI normally

Login should optionally preserve intended redirect destination.

---

# Login architecture

## Route

* `/login`

## Behaviour

* public access route
* not linked in public nav
* used by internal and operator users
* redirects authenticated users appropriately

## Public-site rule

The public site should not mention the existence of login.

---

# Admin bootstrap script architecture

## Purpose

There must be a script that can set or reset the admin user email and password.

This exists to:

* bootstrap first access
* recover access safely
* reduce operational friction

## Recommended characteristics

The script should:

* live in a clear scripts location
* accept explicit input
* validate email and password presence
* hash password correctly
* create admin if missing
* update admin if existing
* produce clear operator output

## Suggested inputs

Examples:

* email
* password

Optional:

* role
* force reset flag

## Documentation requirements

The script must be documented for:

* local usage
* server usage
* expected environment
* safety considerations

# Content and code ownership boundaries

## Public website

* static-first
* repo-backed content
* no unnecessary API dependency

## Protected product UI

* API-backed
* authenticated
* TanStack Query driven

## Shared design system

Both public and protected routes may use shared UI primitives, but their layouts and navigation should remain distinct.

---

# Acceptance criteria

This content and routing architecture is successful when:

* route boundaries are clear
* homepage content is static-first
* blog is repo-backed
* protected routes live under `/app`
* login is reachable but not advertised
* admin bootstrap path exists
* public navigation remains minimal
* SEO foundations exist for public content
