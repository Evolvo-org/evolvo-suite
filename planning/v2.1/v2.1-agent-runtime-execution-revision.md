# Evolvo v2.1 Agent Runtime Execution Revision

## Purpose

This document defines the v2.1 execution model for AI-backed agents.

It is a follow-up revision to the existing v2 planning pack and the v2.1 planning-model revision.

Its purpose is to make agent execution responsibilities explicit and prevent AI model access from leaking into the hosted control plane.

---

# Summary

Evolvo v2.1 has four active AI-backed agent roles:

- planning
- dev
- review
- release

All of them execute only in `apps/runtime`.

The hosted services do not execute AI models:

- `apps/web` is the operator interface
- `apps/api` is the control plane and source of truth
- `apps/runtime` is the only AI execution surface

The runtime must also own the single flat file that controls which model/provider each active agent role uses.

---

# Core rules

## Rule 1 - Only runtime may execute AI models

Only `apps/runtime` may:

- initialise OpenAI and Codex clients
- load OpenAI and Codex credentials
- call the OpenAI API
- call the Codex SDK
- execute prompts for planning, dev, review, or release
- parse model output directly from providers

This is the hard execution boundary.

## Rule 2 - API must never execute AI models directly

`apps/api` must not:

- import provider SDKs for agent execution
- hold agent-execution credentials unless strictly needed for non-agent features
- call AI providers directly
- perform fallback agent execution

The API may:

- schedule and lease work
- persist agent runs
- store prompt snapshots
- store structured results
- store usage records
- expose agent-run history
- enforce workflow rules

## Rule 3 - Web must never execute AI models directly

`apps/web` must not:

- call AI providers directly
- hold agent-execution secrets
- contain provider SDK logic for agent execution
- bypass the API and runtime to invoke AI work

The web may:

- display agent activity
- display model-routing metadata
- display usage
- trigger API commands that later result in runtime execution

## Rule 4 - Runtime owns model routing

Runtime must own model/provider selection for each active agent role.

There must be one flat config file that acts as the main routing source.

Recommended path:

- `apps/runtime/src/config/agent-model.config.ts`

This file should be the first place an operator looks when changing which model a role uses.

## Rule 5 - Runtime owns usage origination

Because runtime is the only place where model execution happens, runtime must originate model usage reporting.

Runtime must report to the API:

- agent role
- provider
- model
- input tokens
- output tokens
- total tokens
- estimated cost where available
- run id
- project id where relevant

The API stores and aggregates usage, but runtime is the execution source.

# Active agent roles

## Planning
Purpose:
- read product description
- read development plan
- determine scope shape
- expand the supplied plan into epics, tasks, and subtasks
- generate acceptance criteria
- generate ambiguity notes where needed

Execution:
- model execution happens in runtime
- persistence happens through the API

## Dev
Purpose:
- implement approved work
- operate in task-scoped worktrees
- produce code and local artifacts
- run required checks

Execution:
- model execution happens in runtime
- git and worktree actions happen in runtime
- persistence happens through the API

## Review
Purpose:
- reason over task output
- combine model review with quality gate results
- produce pass/fail outcome and feedback

Execution:
- model execution happens in runtime
- quality-gate command execution happens in runtime
- persistence happens through the API

## Release
Purpose:
- reason through merge/release actions
- attempt merge conflict resolution where configured
- generate release notes

Execution:
- model execution happens in runtime
- release and repo actions happen in runtime
- persistence happens through the API

# Runtime execution flow

The intended flow for an AI-backed agent run is:

1. API determines eligible work
2. API creates a lease or run record
3. runtime receives leased work
4. runtime assembles context for the relevant agent role
5. runtime loads model configuration for that role
6. runtime resolves the provider adapter
7. runtime executes the model call
8. runtime processes the result
9. runtime performs any required local repo or worktree actions
10. runtime reports result, logs, artifacts, and usage back to the API

The API remains the source of truth.
The runtime remains the execution engine.

# Flat model-config requirement

## Required file

There must be one flat file in runtime for role-to-model routing.

Recommended path:

- `apps/runtime/src/config/agent-model.config.ts`

Alternative acceptable path:

- `apps/runtime/src/config/model-routing.config.ts`

The important rule is that there is one obvious flat file.

## Required fields

Each active role should be configurable for at least:

- provider
- model
- temperature if supported
- max tokens
- timeout
- retry count
- enabled flag

Optional fields may include:

- fallback model
- reasoning level
- json-output mode
- tool-call mode
- local vs remote execution hint
- cost tier hint

## Example shape

```ts
export const agentModelConfig = {
  planning: {
    provider: 'openai',
    model: '...',
    temperature: 0.1,
    maxTokens: 12000,
    timeoutMs: 120000,
    maxRetries: 1,
    enabled: true,
  },
  dev: {
    provider: 'openai',
    model: '...',
    temperature: 0.1,
    maxTokens: 16000,
    timeoutMs: 180000,
    maxRetries: 1,
    enabled: true,
  },
  review: {
    provider: 'openai',
    model: '...',
    temperature: 0.1,
    maxTokens: 12000,
    timeoutMs: 120000,
    maxRetries: 1,
    enabled: true,
  },
  release: {
    provider: 'openai',
    model: '...',
    temperature: 0.1,
    maxTokens: 12000,
    timeoutMs: 120000,
    maxRetries: 1,
    enabled: true,
  },
} as const
````

The exact values are implementation detail.
The flat-file rule is mandatory.

# Provider adapter rule

Runtime should use provider adapters rather than scattered provider calls.

Recommended runtime structure:

* `apps/runtime/src/config/agent-model.config.ts`
* `apps/runtime/src/providers/`
* `apps/runtime/src/agents/`

The provider layer should abstract:

* OpenAI API
* Codex SDK

This keeps agent role logic clean and makes switching providers much easier.

---

# Prompt and execution ownership

## Runtime owns prompt execution

Prompt assembly may use shared utilities if useful, but final model execution must happen inside runtime.

Runtime should own:

* context assembly
* prompt execution
* structured output parsing
* model error handling
* usage capture

## API ownership boundary

The API may store:

* prompt snapshots
* run metadata
* result summaries
* usage records
* artifacts
* logs

But it must not execute the prompt against a provider.

---

# Secrets rule

All AI execution secrets must live in runtime configuration only.

Examples:

* OpenAI API keys
* Codex credentials

These secrets must not be required by:

* web
* api

unless separately needed for an unrelated non-agent feature.

---

# Startup validation requirement

Runtime should validate at startup that:

* the flat model-config file is valid
* each enabled role has a valid provider/model mapping
* required secrets exist
* unsupported provider/model combinations fail clearly
* disabled roles are handled explicitly

This should fail fast rather than allowing silent misconfiguration.

---

# Scheduler implications

The scheduler and lease engine remain API-owned.

The execution rule is:

* API schedules
* runtime executes

The API should never attempt direct “just run this model now” behaviour.

All agent execution must pass through runtime.

---

# Web implications

The web app may show:

* current role-to-model mapping
* agent run history
* usage metrics
* execution status
* approval status

But the web app must not directly edit provider state in a way that bypasses runtime.

If model configuration becomes UI-editable later, runtime must still be the execution authority and final config consumer.

For v2.1, the runtime flat file is the required source.

---

# Testing implications

The system should protect this boundary with tests.

Recommended tests:

* runtime model-config loading test
* provider resolution test per role
* planning execution through runtime test
* usage reporting test
* architecture check that API does not execute provider SDKs directly
* architecture check that web does not execute provider SDKs directly

---

# Benefits

This revision improves the system by:

* making execution boundaries explicit
* keeping secrets off the hosted control plane
* simplifying provider swapping
* centralising model routing
* making runtime the real execution surface
* reducing architectural drift
* making agent behaviour easier to reason about

# Supersession note

If earlier v2 or v2.1 documents imply that:

* the API may directly execute agents
* the web may directly execute agents
* model routing may be spread across multiple unclear locations

those implications are superseded by this revision.

The active rules are:

* only `apps/runtime` may execute AI models
* runtime owns agent model routing
* there must be one flat runtime model-config file
