# Evolvo v2.1 Planning Model Revision

## Purpose

This document defines the Evolvo v2.1 revision to the planning and execution model.

It is a follow-up revision to the existing v2 planning pack.

The purpose of v2.1 is to simplify backlog creation, remove autonomous idea generation, and make planning explicitly driven by a supplied product description and development plan.

This document should be read as a delta on top of the existing planning files.

---

# Summary of the v2.1 change

Evolvo v2 originally included an idea-generation style inbox flow.

That is now removed.

Evolvo v2.1 starts from:

- a product description
- a development plan

The development plan may represent:
- the whole product
- a feature
- a scoped delivery slice

The planning agent must expand the supplied development plan into the full executable backlog structure:

- epics
- tasks
- subtasks

That generated structure is then reviewed and approved by the operator.

Only after approval can execution begin.

---

# Core v2.1 rules

## Rule 1 - Remove idea generation

The system does not independently generate candidate ideas for a project.

There is no autonomous idea backlog creation stage.

There is no inbox discovery loop.

There is no project analysis agent whose purpose is to invent work outside the supplied plan.

## Rule 2 - Planning starts from supplied inputs

Planning starts only from:

- product description
- development plan

These two inputs are the source material for backlog generation.

## Rule 3 - Development plan is the planning scope

The planning agent must treat the supplied development plan as the intended scope.

That scope may represent:

- the whole product
- a new major feature
- a refinement or expansion of an existing product
- a delivery slice or milestone

The planning agent may refine and decompose the plan, but must not replace it with a different plan.

## Rule 4 - Planning agent expands the whole supplied plan

The planning agent must build the full backlog structure for the supplied development plan.

That means producing:

- epics
- tasks
- subtasks
- acceptance criteria where applicable

The planning agent is not only selecting a single next task.
It is expanding the whole planned scope into executable structure.

## Rule 5 - Approval gates execution

The generated planning structure must be reviewed and approved before execution starts.

Approval confirms that:

- the generated epic/task/subtask hierarchy matches the intended scope
- the decomposition is acceptable
- the system may begin execution

Only approved planned work may move into execution states.

## Rule 6 - Execution begins after planning approval

After approval:

- approved tasks or subtasks may move to `Ready for dev`
- the dev agent may begin implementation
- review and release continue as normal

# Workflow changes

## Removed concepts

The following concepts are removed in v2.1:

- Inbox agent
- Inbox column
- autonomous idea generation
- idea triage as a first-class workflow stage
- project-wide round-robin idea discovery

## Retained concepts

The following concepts remain:

- planning hierarchy
- strict workflow states
- dev agent
- review agent
- release agent
- worktree-per-task model
- approval and human intervention flows
- per-project queue limits
- scheduler fairness and leasing

---

# Updated workflow columns

The active execution columns for v2.1 are:

- Planning
- Ready for dev
- In dev
- Ready for review
- In review
- Ready for release
- Requires human intervention
- Released

There is no `Inbox` column in v2.1.

---

# Updated agent model

## Planning agent
One global planning agent across all projects.

Responsibilities:
- read product description
- read development plan
- determine whether the plan describes a whole product build or a feature build
- expand the supplied scope into epics, tasks, and subtasks
- generate acceptance criteria where appropriate
- persist generated planning structure
- wait for approval before execution begins

The planning agent does not invent independent backlog ideas.

## Dev agent
One dev lane per project.

Responsibilities:
- watch approved work in `Ready for dev`
- lease work safely
- implement in a task-scoped worktree
- run required checks
- move work to `Ready for review`

## Review agent
One global review agent.

Responsibilities:
- process `Ready for review`
- move to `In review`
- validate quality gates and acceptance criteria
- move work back to `Ready for dev` with comments on failure
- move work to `Ready for release` on pass

## Release agent
One global release agent.

Responsibilities:
- process `Ready for release`
- attempt merge conflict resolution within configured limits
- merge to main
- create tag
- generate release notes
- escalate to `Requires human intervention` when blocked

# Updated planning model

## Required project planning inputs

A project should now require:

- project name
- repository configuration
- product description
- development plan

The development plan is no longer optional for the v2.1 planning flow.

If the operator wants the system to build a whole product, the development plan should describe the whole product delivery.

If the operator wants the system to build a feature, the development plan should describe that feature delivery.

## Planning output

The planning output should include:

- interpreted scope summary
- epic hierarchy
- task hierarchy
- subtask hierarchy
- acceptance criteria
- any dependency notes
- any identified ambiguity or operator review flags

---

# Approval model

## Approval target

Approval is applied to the generated planning structure, not only a single task.

The operator should be able to review:

- plan summary
- epics
- tasks
- subtasks
- acceptance criteria

## Approval result

Once approved:

- planning becomes execution-eligible
- items may move from `Planning` to `Ready for dev`
- the dev lane may begin working through the approved scope

Without approval, work remains in `Planning`.

---

# Scheduler implications

The scheduler no longer needs to allocate cycles to idea generation.

The scheduler should instead support:

- planning expansion runs
- approval-aware promotion from `Planning` to `Ready for dev`
- normal dev/review/release orchestration
- per-project queue limit enforcement

This should reduce wasted planning cycles and make execution more deterministic.

---

# Domain implications

## Remove or deprecate
The following concepts should be removed or deprecated from the active domain model:

- Inbox work item type if only used for idea intake
- Inbox queue state
- inbox generation services
- idea acceptance/rejection flows
- inbox round-robin scheduling logic

## Retain and strengthen
The following concepts should remain central:

- ProductSpec
- DevelopmentPlan
- PlanVersion
- Epic
- WorkItem
- AcceptanceCriterion
- WorkItemStateTransition
- AgentRun
- HumanInterventionCase

---

# UI implications

The web UI should be revised to reflect the v2.1 planning model.

## Remove
- Inbox page or column if present
- idea triage controls
- idea acceptance/rejection controls
- inbox agent activity summaries

## Add or emphasise
- plan expansion action
- plan approval flow
- planning hierarchy review before execution
- clear distinction between unapproved planning and approved execution-ready work

---

# Runtime implications

The local runtime is not materially changed by v2.1.

The runtime still:
- receives leased execution work
- implements tasks
- reviews/reports results
- participates in release flow

The main change is upstream:
the runtime only begins work after planning approval has occurred.

---

# Benefits of the v2.1 revision

This revision improves the system by:

- reducing autonomous scope drift
- making planning more deterministic
- aligning execution to explicit operator intent
- removing a weaker speculative planning stage
- making approval more meaningful
- simplifying orchestration and scheduling
- improving predictability for product and feature delivery
